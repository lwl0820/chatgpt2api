## Context

当前选图页在切换候选图时，会优先尝试缩略图资源；当该候选没有缩略图时，加载链路会先经历一次失败请求，再回退到原图。这种行为在用户反复切换候选时会持续产生 404 请求，增加网络噪音和切换延迟。

本次优化只针对“已经确认没有缩略图的候选”这一场景，不改变正常存在缩略图时的展示顺序，也不改动全局图片管理的资源语义。

## Goals / Non-Goals

**Goals:**
- 让已经确认没有缩略图的候选在后续切换时直接加载原图。
- 避免对同一候选重复发起会返回 404 的缩略图请求。
- 保持有缩略图候选的现有行为不变。
- 让这一判断在候选切换和会话恢复后仍然可用。

**Non-Goals:**
- 不修改缩略图生成策略。
- 不修改后端图片存储或下载接口。
- 不改变候选保留、丢弃、下载、预加载的业务语义。
- 不把这次优化扩展到全局图片管理的其他页面。

## Decisions

1. **按候选级别缓存缩略图可用性状态。**
   为每个候选记录一个轻量状态标记，例如 `unknown / available / missing`，在同一选图会话内复用。这样可以在候选被再次选中时直接跳过缩略图探测。
   - 备选方案：只在组件内存中缓存。这个方案实现更快，但页面刷新后会丢失状态，无法覆盖恢复场景。
   - 备选方案：按全局图片路径共享缓存。这个方案复用率更高，但会把选图会话里的判定扩散到其他页面和场景，边界更难控制。

2. **在决定请求资源前先检查缓存状态。**
   如果候选已经被标记为 `missing`，资源解析逻辑直接返回原图地址，不再构造缩略图请求，也不会等待 404 再回退。
   - 备选方案：继续先试缩略图，再回退原图。这个方案保留了现状中的额外请求，无法解决当前问题。
   - 备选方案：改成额外的 HEAD/探测请求。这个方案只是把 404 请求换成另一种探测请求，仍然增加一次网络往返。

3. **在首次确认缺失时写回候选状态。**
   当候选第一次因为缩略图缺失而进入原图路径时，将该候选标记为 `missing`，用于后续复用。
   - 备选方案：只依赖运行时推断，不写回状态。这个方案会让下一次切换仍然重新探测，无法真正减少重复请求。

4. **候选身份变化时清理旧状态。**
   如果候选被重新生成、路径变化或被新资源替换，需要清除旧的缩略图可用性标记，避免把旧图片的缺失判断错误地带到新图片上。
   - 备选方案：永不清理。这个方案实现简单，但容易产生 stale cache 问题。

5. **不引入后端契约变更。**
   这个优化可以完全由前端/本地会话状态完成，不需要后端先暴露缩略图存在性元数据。
   - 备选方案：让后端直接返回 thumbnail presence。这个方案更明确，但会扩大接口面，且对当前优化来说收益不成比例。

## Risks / Trade-offs

- [Risk] 首次遇到确实没有缩略图的候选时，仍然可能产生一次失败请求。 → [Mitigation] 只缓存一次即可消除后续重复请求，已经能覆盖当前问题的主要成本。
- [Risk] 候选资源如果被替换但缓存未清理，可能误判为没有缩略图。 → [Mitigation] 在候选身份或资源路径变化时主动清除状态。
- [Risk] 额外的候选状态字段会增加少量本地持久化复杂度。 → [Mitigation] 使用可选字段和默认 `unknown`，保持旧数据兼容。

## Migration Plan

1. 为候选状态增加一个可选的缩略图可用性字段，默认值为 `unknown`。
2. 更新候选资源解析逻辑，优先读取该字段并在 `missing` 时直接返回原图。
3. 在缩略图缺失回退到原图的路径上写回 `missing` 状态。
4. 保持旧会话可直接读取，新字段缺失时沿用现有回退逻辑。
5. 如需回滚，只需忽略新增状态字段并恢复原有缩略图探测顺序。
