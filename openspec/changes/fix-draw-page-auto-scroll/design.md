## Context

画图页面使用 `resultsViewportRef` 管理对话结果区域滚动。当前实现会在 `selectedConversation`、`selectedConversation?.updatedAt` 或 `turns.length` 变化时平滑滚动到底部。图片任务运行期间，队列逻辑每隔约 2 秒轮询任务状态，并在 `applyTasks` 或错误处理时更新会话的 `updatedAt`。因此即使只是同一条任务的状态刷新，也会触发自动滚动，覆盖用户手动停留在上方内容的意图。

本变更只调整前端滚动策略和结果视口辅助控件，不改变图片任务、会话持久化或 API 数据模型。

## Goals / Non-Goals

**Goals:**
- 仅在用户提交新内容或新任务出现在对话中时主动滚到底部。
- 当任务处于处理中、轮询更新、完成或失败时，不自动改变用户滚动位置。
- 提供“回到底部”悬浮按钮，让用户按需查看最新内容。
- 保持提交新画图请求、重试、重新生成、继续编辑等新任务创建场景的自动滚动体验。

**Non-Goals:**
- 不重构图片任务队列、轮询频率或后端任务接口。
- 不改变会话排序和 `updatedAt` 持久化语义。
- 不新增全局滚动管理框架或外部依赖。
- 不基于“接近底部”阈值自动跟随处理中任务更新。

## Decisions

### 只对用户发起的新内容强制滚动

自动滚动只应响应用户明确发起的新内容事件，例如提交新的画图请求、重试、重新生成或继续编辑后新任务出现在对话中。任务进入执行阶段后的轮询状态变化、成功结果写入和失败状态写入都不得自动滚动。

备选方案是使用“接近底部”阈值让底部用户自动跟随任务更新，但这仍可能在执行中产生非用户主动触发的滚动。当前选择更可预测：任务执行期间完全不干预，用户需要查看底部时点击按钮。

### 用显式滚动意图替代状态差异猜测

实现上应避免依赖任意 `selectedConversation` 对象引用、`updatedAt` 或轮询造成的状态差异来推断滚动意图。用户提交、重试、重新生成、继续编辑等入口在创建新任务或新轮次时标记一次“应滚到底部”的意图，由结果视口 effect 消费该意图后执行滚动。

备选方案是在视口 effect 中比较会话 ID、轮次数量或图片数量，但这些信号容易与任务补偿提交、状态同步和历史加载混在一起。显式意图更贴近用户动作，也更容易验证不会被轮询误触发。

### 增加回到底部悬浮按钮

结果视口应在用户不在底部时显示一个轻量的悬浮按钮。按钮点击后滚动到底部，之后不改变任务执行期间“不自动滚动”的规则。按钮只作为用户主动操作入口，不作为自动跟随机制。

备选方案是不提供按钮，仅依赖用户手动滚动到底部；但在长会话中查找最新结果成本较高，悬浮按钮能补足执行中不自动滚动带来的可达性问题。

## Risks / Trade-offs

- [Risk] 某些新任务入口未标记滚动意图 → 覆盖提交、重试、重新生成、继续编辑等入口，并在任务清单中逐项验证。
- [Risk] 悬浮按钮显示状态与实际滚动位置不同步 → 在结果视口滚动事件中维护按钮可见性，并在点击滚到底部后更新状态。
- [Risk] 执行中完全不自动滚动可能让等待底部结果的用户少一步操作 → 通过醒目的“回到底部”悬浮按钮提供明确的主动入口。

## Migration Plan

无需数据迁移。变更可作为前端代码更新直接发布。若出现回归，可回滚到当前自动滚动 effect，但会恢复原有抢滚问题。
