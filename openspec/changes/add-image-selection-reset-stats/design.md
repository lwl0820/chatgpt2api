## 背景

选图页会在会话头部和沉浸模式中展示队列、保留、丢弃、跳过等统计。旧实现会从候选状态实时派生保留、丢弃和跳过数量，重置时依赖时间边界过滤历史候选。由于前后端时间格式不一致，历史失败候选可能仍被计入跳过统计。现在需要把保留、丢弃和跳过改为会话级持久化统计值，让重置统计直接清零这些数值。

## 目标 / 非目标

**目标：**
- 在当前选图会话中提供可见的“重置统计”操作。
- 将保留、丢弃、跳过统计作为会话级持久化数值保存。
- 重置后直接把保留、丢弃和跳过统计清零，后续新的保留、丢弃和失败跳过继续递增对应统计。
- 保持候选图、图片文件、会话配置、运行状态和队列补齐行为不受影响。

**非目标：**
- 不删除或恢复任何本地图片文件。
- 不清空候选队列、保留归档或丢弃记录。
- 不改变会话列表的排序语义；重置统计不应更新用于排序的 `updatedAt`。
- 不新增后端公开 API 或外部依赖。

## 决策

- 使用会话级 `stats` 字段保存保留、丢弃、跳过计数，而不是从候选状态实时派生。
  - 原因：重置统计是用户可见的累计数值操作，直接持久化数值能避免时间格式、历史候选字段缺失和状态回放导致的误计数。
  - 备选方案：继续使用 `statsResetAt` 时间边界过滤候选；实现较少，但会受历史数据和前后端时间格式影响，已经导致跳过计数无法稳定重置。
- 保留、丢弃由前端决策事件递增，撤销时递减对应统计。
  - 原因：这些操作由用户显式触发，前端知道候选是否真正从 `ready` 进入最终状态，可以避免重复点击造成重复统计。
  - 备选方案：保存时扫描候选状态差异；会让 UI 操作和同步逻辑耦合更重。
- 跳过由后端队列服务在候选首次进入 `error` 状态时递增。
  - 原因：失败/跳过主要由后台任务同步产生，后端最清楚候选是否从非错误状态转为错误状态。
  - 备选方案：前端继续按 `error` 候选数量展示；无法可靠支持重置后清零。
- 重置统计通过当前会话的保存逻辑持久化，但不改变会话排序时间。
  - 原因：既满足刷新后仍保持重置结果，又符合既有要求中“生图任务和候选状态变化不更新排序时间”的语义。
  - 备选方案：复用配置保存并更新 `updatedAt`；实现更直接，但会错误影响会话列表顺序。

## 风险 / 权衡

- [风险] 候选状态数量和会话级统计值可能不一致。 → 缓解：把 UI 文案保持为“统计”，而不是“当前候选数量”，并只在明确事件发生时更新统计。
- [风险] 重置统计后，保留归档数量和展示的保留统计可能暂时不一致。 → 缓解：UI 文案和行为保持“统计重置”的含义，明确不清空归档或候选状态。
- [风险] 后端重复同步同一个失败候选时可能重复增加跳过统计。 → 缓解：保存候选前比较最新会话里的候选状态，只在候选从非 `error` 变成 `error` 时递增。

## 迁移计划

- 不做旧重置字段迁移；未带 `stats` 的会话按 0 初始化。
- 发布后验证新会话能正确记录保留、丢弃、跳过统计，并且重置后刷新仍保持 0。
- 若需要回滚，只需恢复统计计算逻辑和重置入口，现有候选状态与文件不受影响。
