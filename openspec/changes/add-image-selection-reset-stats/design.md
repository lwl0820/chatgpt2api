## 背景

选图页会在会话头部和沉浸模式中展示队列、保留、丢弃、跳过等统计。现有统计主要从候选状态派生，保留和丢弃来自候选决策，跳过对应失败候选。用户需要在不清理候选、图片文件或会话配置的情况下，将这些统计清零，并从之后的新操作重新计数。

## 目标 / 非目标

**目标：**
- 在当前选图会话中提供可见的“重置统计”操作。
- 将保留、丢弃、跳过统计重置为 0，并持久化该重置状态。
- 重置后，后续新的保留、丢弃和失败跳过继续计入统计。
- 保持候选图、图片文件、会话配置、运行状态和队列补齐行为不受影响。

**非目标：**
- 不删除或恢复任何本地图片文件。
- 不清空候选队列、保留归档或丢弃记录。
- 不改变会话列表的排序语义；重置统计不应更新用于排序的 `updatedAt`。
- 不新增后端公开 API 或外部依赖。

## 决策

- 使用会话级统计重置标记来计算展示统计，而不是批量改写候选状态。
  - 原因：候选状态仍是选图工作流和图片管理过滤的事实来源，重置展示统计不应破坏保留、丢弃、跳过记录。
  - 备选方案：将所有 kept/discarded/error 候选改回 ready 或直接移除；这会改变选图历史和图片管理过滤结果，超出需求。
- 让统一的统计计算函数负责处理重置后的展示值。
  - 原因：页面多个位置复用同一统计结果，集中处理可避免普通模式、沉浸模式和会话列表出现不一致。
  - 备选方案：在 UI 层分别扣减或清零；容易遗漏调用点，也会让测试覆盖更分散。
- 重置统计通过当前会话的保存逻辑持久化，但不改变会话排序时间。
  - 原因：既满足刷新后仍保持重置结果，又符合既有要求中“生图任务和候选状态变化不更新排序时间”的语义。
  - 备选方案：复用配置保存并更新 `updatedAt`；实现更直接，但会错误影响会话列表顺序。
- 跳过统计继续对应失败候选，重置后只统计重置点之后进入失败状态的候选。
  - 原因：用户看到的“跳过”仍保持当前语义，只增加清零后的计数边界。
  - 备选方案：把跳过改成独立事件计数；会引入更大的数据模型迁移成本。

## 风险 / 权衡

- [风险] 现有历史会话缺少重置字段或失败时间字段，可能导致首次实现时无法区分重置前后的旧失败候选。 → 缓解：为缺失字段提供兼容默认值，并在候选进入失败状态时记录可用于统计边界的时间。
- [风险] 重置统计后，保留归档数量和展示的保留统计可能暂时不一致。 → 缓解：UI 文案和行为保持“统计重置”的含义，明确不清空归档或候选状态。
- [风险] 撤销重置前的决策后再重新决策时，统计可能因时间边界处理出错。 → 缓解：让决策操作更新候选的决策时间，并通过单元测试覆盖重置后的保留、丢弃和撤销路径。

## 迁移计划

- 不需要数据迁移，旧会话按默认重置边界处理即可。
- 发布后先验证新会话和历史会话都能正确显示重置后的保留、丢弃、跳过统计。
- 若需要回滚，只需恢复统计计算逻辑和重置入口，现有候选状态与文件不受影响。
