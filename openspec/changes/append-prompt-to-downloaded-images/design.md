## Context

当前项目存在多个图片下载入口：画图会话结果在前端从 `b64_json` 或 URL 构造 Blob 下载，图片预览弹窗直接下载当前图片地址，图片选择页面下载候选图并随后标记为保留，图片管理器通过后端接口下载单图或 zip 包。

提示词主要存在于画图会话的 turn 数据和图片选择 session 数据中；部分图片管理器文件可能无法可靠反查提示词。用户明确要求提示词写在图片文件数据最后，而不是追加到文件名，并要求通过设置页面开关控制该特性，默认关闭。

## Goals / Non-Goals

**Goals:**

- 设置页面提供默认关闭的开关，用于控制图片下载是否追加提示词文本载荷。
- 开关启用后，所有能获得提示词的图片下载都在下载文件的二进制数据末尾追加提示词文本载荷。
- 开关关闭时，所有图片下载保持既有文件内容。
- 保持现有下载文件名、图片格式扩展名和用户交互语义不变。
- 让追加载荷具备稳定边界，方便后续工具从文件尾部识别并提取提示词。
- 对没有提示词的下载保持现状，避免写入空提示词或误导性元数据。

**Non-Goals:**

- 不修改原始存储图片文件，只修改下载给用户的文件内容或 zip 条目内容。
- 不把提示词写入 EXIF、PNG chunk 或其他格式专用元数据结构。
- 不提供提示词提取 UI 或命令行工具。
- 不尝试为历史图片建立完整的文件到提示词反向索引。

## Decisions

- 新增一个持久化布尔配置项，默认值为 `false`。设置 API 返回和保存该字段，设置页面以开关形式展示；缺失旧配置按关闭处理，避免行为在升级后自动改变。备选方案是默认开启以最大化可追溯性，但这会改变现有下载文件内容，不符合用户要求。
- 使用统一的尾部文本载荷格式。下载时将原始图片字节后追加 UTF-8 文本，建议格式为 `\n\n-- chatgpt2api prompt --\n<prompt>\n-- end chatgpt2api prompt --\n`。这样不依赖具体图片格式，也满足“文件数据最后”的要求。备选方案是写入 EXIF 或 PNG 文本块，但会引入格式分支，且对 WebP、远程 Blob 和 zip 条目处理更复杂。
- 在前端提供共享下载辅助函数。画图结果、预览弹窗和图片选择页面都可以复用同一逻辑：先检查设置开关，再把 data URL、base64 或远程 URL 转成 Blob，仅在开关启用且提示词非空时在 Blob 尾部追加提示词后触发下载。备选方案是在各组件内分别拼接 Blob，但会增加格式和错误处理不一致的风险。
- 后端图片管理器下载只在设置开关启用且能确定提示词时追加载荷。单图响应和 zip 条目可以在服务端读取原图片字节并追加提示词；如果开关关闭、提示词为空或当前数据模型无法把文件路径可靠映射到提示词，则保持原文件内容。备选方案是强制为所有管理器下载写入空载荷，但这会制造无意义元数据。
- 图片选择下载使用 session prompt。候选图本身没有独立提示词字段时，下载当前候选图应在开关启用时使用所属选择 session 的提示词，并继续沿用现有“下载等同保留并前进”的决策流程。

## Risks / Trade-offs

- 部分图片查看器可能忽略尾部额外字节，但少数严格解析工具可能认为文件包含附加数据 -> 仅对下载副本追加，不修改服务端原图；保留原扩展名和 MIME 类型。
- 追加纯文本不是标准图片元数据 -> 这是为了满足跨格式和“文件数据末尾”的明确要求；通过固定边界降低提取歧义。
- 开关状态需要前后端一致 -> 前端下载读取设置缓存或加载结果，后端图片管理器下载读取持久化配置，以同一默认关闭语义处理缺失字段。
- 图片管理器无法覆盖所有历史文件 -> 无可靠提示词来源时不追加，避免把错误提示词写进下载文件。
- 大批量 zip 下载会增加内存复制成本 -> 只在写入 zip 条目前追加必要字节，避免改写磁盘文件。

## Migration Plan

无需迁移历史图片。实现发布后，缺失新配置的环境按关闭处理；管理员在设置页面手动开启后，新下载的图片副本才开始携带提示词尾部载荷。已有本地图片文件和历史会话数据保持不变。若需要回滚，关闭设置或移除下载时追加逻辑即可，原始存储文件不受影响。

## Open Questions

- 图片管理器中的历史文件是否已有稳定的路径到提示词映射来源；若没有，本次实现仅覆盖可从现有会话或选择 session 数据匹配到的文件。
