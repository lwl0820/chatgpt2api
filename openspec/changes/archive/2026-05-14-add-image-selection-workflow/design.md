## Context

当前画图页面通过 `web/src/app/image/page.tsx` 创建图片任务，并将画图会话持久化到浏览器本地 IndexedDB。图片任务由 `/api/image-tasks/generations` 提交、`/api/image-tasks` 轮询，成功结果会返回已保存到本地图片目录的 URL。图片管理页面从 `/api/images` 拉取本地图片文件列表，并支持标签、日期、下载和物理删除等管理能力。

选图工作流需要围绕同一提示词持续生成候选图，并让用户快速保留或丢弃候选。这里的“丢弃”与现有画图会话删除保持一致：只改变前端会话状态，不删除本地图片文件。图片管理需要能够按选图会话筛选已保留图片，因此选图会话必须记录每张候选图对应的图片相对路径，供图片管理列表与本地归档关系匹配。

## Goals / Non-Goals

**Goals:**
- 新增独立选图页面，支持输入提示词后持续补齐候选队列。
- 以大图为核心展示当前候选，并同时支持方向键和按钮进行保留/丢弃。
- 候选生成错误自动跳过，不要求用户处理失败卡片。
- 保留和丢弃都只改变选图会话状态，不物理删除本地图片文件。
- 图片管理页面支持按选图会话筛选展示该会话保留的图片。
- 页面刷新或重新打开后恢复选图会话状态，但不自动继续提交新的生成任务。
- 创建选图会话时提供本次队列长度和连续错误暂停阈值配置。

**Non-Goals:**
- 不重构现有画图页面的会话模型或图片任务 API。
- 不引入后端数据库保存选图会话；选图归档关系与现有画图历史一致，保存在浏览器本地。
- 不在选图丢弃时调用 `/api/images/delete` 或删除物理文件。
- 不改变现有图片管理页的全局图片列表、下载、标签和物理删除语义。
- 不支持页面关闭后后台继续补齐队列。

## Decisions

### 独立选图页面而非塞进画图页

选图页面应作为 `/image-select` 独立入口实现。现有 `/image` 面向“提交一次，生成固定张数，进入对话历史”的流程，而选图需要持续补齐队列、大图审片、快捷键决策和暂停恢复状态。拆分页面可以避免把持续生成状态机与现有画图对话、编辑参考图、重试、滚动策略耦合在一起。

备选方案是在画图页增加选图模式，但会让页面已有状态继续膨胀，并且选图的交互重心与对话式画图不同。独立页面更容易验证和回滚。

### 选图会话保存在前端 IndexedDB

选图会话应新增独立本地 store，例如 `image-selection-sessions`。会话记录提示词、尺寸、队列上限、连续错误暂停阈值、运行状态和候选列表。候选记录 `taskId`、状态、图片 URL、图片相对路径、错误信息和决策时间。

备选方案是在后端增加选图会话存储，但当前画图会话历史已经是浏览器本地语义，引入后端持久化会带来身份归属、迁移和跨设备同步问题。本变更先与现有前端历史模型保持一致。

### 候选队列按 loading + ready 补齐

选图运行时只把 `loading` 和 `ready` 候选计入队列占位。`kept`、`discarded` 和 `error` 候选不占队列位。每次任务成功、失败、保留或丢弃后重新计算空位；当空位大于 0 且会话处于运行状态时，提交新的生成任务直到队列达到配置上限。

备选方案是一次提交固定数量后等待用户处理完一批再提交下一批，但这会产生明显等待间隙，不符合“不断生成新的图直到填满队列”的体验。补位模型能在用户筛选时保持候选池持续可用。

### 错误自动跳过并需要防止无限失败

单个候选任务失败时标记为 `error`，不展示为当前候选，也不占队列位。页面可以展示累计跳过数量或最近错误摘要，但用户不需要手动删除失败候选。为避免上游持续失败时无限消耗请求，应在创建选图会话时设置连续失败保护阈值：当连续失败达到该阈值时暂停选图会话并提示用户手动继续。

备选方案是像画图页一样展示失败卡片和重试按钮，但选图目标是快速筛选成功图片，失败卡片会打断审片节奏。

### 保留/丢弃是会话状态，不是文件操作

保留将候选状态变更为 `kept`，并让其进入当前选图会话归档；丢弃将状态变更为 `discarded`。两者都不删除 `data/images` 中的文件，也不调用图片管理删除接口。这样普通用户选图不会获得物理删除权限，也避免误删影响全局图片管理。

备选方案是丢弃时物理删除文件，但现有图片生成结果可能同时被图片管理、画图历史或其他本地引用使用，缺少后端所有权信息时删除风险过高。

### 图片管理按本地选图会话筛选

图片管理页继续从 `/api/images` 获取全部图片文件，再读取本地选图会话，把所选会话的 `kept` 候选相对路径作为筛选集合。启用选图会话筛选时，仅展示 `ManagedImage.rel` 命中该集合的图片。筛选关系是浏览器本地语义；换浏览器或清空本地数据后，物理图片仍存在，但选图归档关系不可用。

备选方案是在后端图片 metadata 中写入选图会话标签，但会将本地前端会话扩展为后端跨用户数据模型，超出本次变更范围。

### 刷新后恢复状态但不自动继续生成

页面加载时应恢复选图会话和已提交候选任务的状态，并可轮询已有 `loading` 任务完成情况；但不得自动提交新的任务补齐队列。会话应进入可继续的暂停状态，用户点击“继续选图”后才恢复自动补位。

备选方案是刷新后自动继续补队列，但这可能在用户无意打开页面时消耗图片额度。手动继续更可控。

## Risks / Trade-offs

- [Risk] 选图归档关系保存在浏览器本地，跨浏览器不可见 → 与现有画图历史保持一致，并在设计中明确图片文件仍可在全局图片管理中查看。
- [Risk] 从图片 URL 解析相对路径失败会导致图片管理筛选漏图 → 生成候选时应统一提取 `/images/` 后的相对路径，并在缺失时保留 URL 但不纳入会话筛选集合。
- [Risk] 持续生成可能快速消耗额度 → 每个会话创建时配置队列长度和连续失败阈值，刷新后默认暂停，并提供手动暂停入口。
- [Risk] 键盘快捷键可能影响输入框编辑 → 仅当焦点不在文本输入控件中时处理方向键决策。
- [Risk] 图片管理筛选与标签/日期筛选叠加可能让列表为空 → UI 应显示当前筛选条件，并提供清除选图会话筛选的入口。

## Migration Plan

无需后端数据迁移。新增前端本地 store 初始为空，不影响已有画图会话和图片管理数据。旧选图会话若缺少连续错误阈值，前端归一化时使用默认值。

若发布后需要回滚，可移除导航入口和选图页面；已生成的图片文件仍保留在本地图片目录，已写入浏览器本地的选图会话数据不会影响现有功能。
