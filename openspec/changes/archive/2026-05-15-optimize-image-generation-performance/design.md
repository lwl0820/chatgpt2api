## 背景

这个项目里的生图逻辑不只是简单转发上游请求：成功结果会落盘，保留期清理会扫描图片目录，缩略图会按需生成，而设置界面也已经持久化了多个图片相关布尔项。在高负载服务器上，重复的文件系统扫描和缩略图处理即使在上游调用空闲时，也足以占满后端 CPU。

这次变更的目标很窄：在不改变生图语义、任务调度和保留策略的前提下，降低单张图片的本地 CPU 开销。

## 目标 / 非目标

**目标：**
- 把自动图片保留期清理从图片保存热路径中移除。
- 增加一个可持久化的管理员缩略图开关。
- 在开关关闭时，避免创建新的缩略图文件。
- 确认图片管理、日志和选图界面已有的原图回退能力，不新增后端响应分支。
- 保留现有保留策略语义，以及对调用方可见的缩略图 URL 形态。

**非目标：**
- 重设计图片保留策略或调度策略。
- 调整图片任务并发行为。
- 调整缩略图响应策略、缩略图清理策略，或围绕新的缩略图 API 重写图片管理和日志界面。
- 在关闭功能时自动删除所有已有缩略图文件。

## 决策

1. 新增一个名为 `image_thumbnail_generation` 的布尔配置项，默认开启。
   - 这符合现有配置模型，图片相关行为本来就以简单布尔值和数字保存在 `config` 中，并映射到 settings store。
   - 备选方案是新增独立表或特殊运行时开关，但当前配置系统已经覆盖这类设置，再引入新的持久化路径只会增加复杂度。

2. 从生成结果保存路径中移除 `cleanup_old_images()` 调用。
   - 真正昂贵的是同步递归文件系统扫描。把它从热路径移除，能直接降低最频繁的写入型流程 CPU 消耗。
   - 备选方案是挪到后台定时任务或缓存扫描结果，但这仍会带来周期性的文件系统开销，还会引入新的调度状态，收益不高。

3. 保持 `/image-thumbnails/*` 路由、`thumbnail_url` 合同和当前响应行为不变。
   - 这个开关只控制是否创建缩略图，不改变已有缩略图清理和响应逻辑。
   - 现有前端组件已经具备缩略图加载失败后回退原图的能力，因此无需为了这个开关新增后端响应分支。

4. 只让缩略图创建受新设置控制。
   - 关闭时，`ensure_thumbnail()` 不再创建新缩略图；已有缩略图、缩略图清理和缩略图响应保持现有行为。
   - 备选方案是让清理和响应也受同一个设置控制，但这会扩大变更范围，并且不符合“只控制创建”的需求。

5. 只在显式的非热路径里保留清理。
   - 启动、列表和维护路径里的现有清理点继续保留，保留策略本身仍然存在。
   - 这次改动只从最敏感的生成结果保存路径中移除清理。

## 风险 / 权衡

- [关闭缩略图后，首次请求不存在的缩略图会回退到原图] → 这是已有前端行为，能减少后端生成缩略图的 CPU 开销，但可能增加客户端加载原图的带宽。
- [关闭功能后，已有缩略图文件仍留在磁盘上] → 这是可接受的非破坏性回滚状态；管理员如果想回收磁盘空间，可以之后再手动清理。
- [缩略图响应不跟随开关变化] → 这是刻意设计，避免为了配置项改变已有接口行为；若已有缩略图存在，仍可正常返回。
- [某些显式维护路径里仍然存在清理] → 这是刻意保留的；目标是移出保存热路径，而不是改变保留策略语义。

## 迁移计划

1. 新增 `image_thumbnail_generation` 配置字段，默认开启，保证已有部署继续保持当前行为。
2. 调整后端图片保存路径，跳过保留期清理。
3. 调整缩略图创建逻辑，在关闭开关时不再生成新缩略图；缩略图清理和响应保持现状。
4. 给 settings UI/store 增加这个开关的读写链路。
5. 无需数据迁移；没有新键的旧配置继续视为开启。
6. 回滚方式：回退代码路径改动即可。持久化的配置键即使保留也无害，已有缩略图文件也可以继续留在磁盘上。

## 未决问题

- 本次没有未决问题。后续是否要增加一个显式“清理缩略图”动作，刻意不在这次范围内。
